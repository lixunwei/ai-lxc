# LXC 核心生命周期深度分析：启动引擎

## 1. 概述

LXC 的容器启动引擎位于 `src/lxc/start.c`，是整个系统最核心的代码。
启动过程通过三层函数实现：

```
__lxc_start()          top entry:init -> spawn -> monitor -> cleanup
  +-- lxc_init()       pre-spawn prep:signals, cgroup, hooks, security
  +-- lxc_spawn()      core spawn:clone3() + parent/child sync
  +-- lxc_poll()       monitor main loop:signals, cmds, console
```

## 4. lxc_spawn() — 核心生成

**位置**：`start.c:1834-2170`

这是启动引擎中最复杂的函数，负责创建容器进程并通过同步协议协调父子进程。

### 4.1 预 clone 设置

```
lxc_spawn(handler)
  |
  +- 1. create sync socketpair
  |      lxc_sync_init(handler)                    [1848-1849]
  |
  +- 2. create data socketpair
  |      socketpair(AF_UNIX, SOCK_STREAM)          [1851-1856]
  |
  +- 3. prepare gateway addr                              [1858-1864]
  |
  +- 4. create container cgroup
         cgroup_ops->payload_create()              [1866-1869]
```

### 4.2 clone3() 系统调用

两种路径：

#### 继承命名空间路径

当需要共享其他容器的命名空间时（`--share-net` 等）：

```

  +-- setns(nsfd, ...)                            // enter inherited ns
  +-- clone(CLONE_PARENT | CLONE_PIDFD)           // with CLONE_PARENT
```

#### 直接 clone3 路径

正常启动路径：

```
clone3(&clone_args, size)                         [1896-1934]
  +-- set clone_args:
  |     .flags = handler->clone_flags             // CLONE_NEW* set
  |     .pidfd = &handler->pidfd
  |     .exit_signal = SIGCHLD
  |     .cgroup = cgroup_fd                       // CLONE_INTO_CGROUP(optional)
  |
  +-- try CLONE_ARGS_SIZE_VER2                   [1911-1915]
  |     supports CLONE_INTO_CGROUP

  +-- fallback CLONE_ARGS_SIZE_VER0                   [1917-1921]

```

## 6. lxc_poll() — 监控主循环

**位置**：`start.c:763-847`

容器启动后，父进程（monitor）进入主循环：

```
lxc_poll(handler, post_start_fn)
  |                                              [763-847]
  +-- create main loop
  |     lxc_mainloop_open(descr)                   [772-778]
  |
  +-- register handlers:
  |     +-- signal handler(SIGCHLD, etc.)               [786-793]
  |     +-- seccomp proxy(if enabled)               [795-799]
  |     +-- console I/O handler                      [801-807]
  |     +-- cmd socket handler                       [809-813]
  |
  +-- run main loop
  |     lxc_mainloop(descr, -1)                    [817-821]
  |     -> epoll/io_uring wait events
  |     -> dispatch to handlers
  |
  +-- loop end(after container exit)
        close all fds                                [822-846]
```

### 6.1 信号处理

```
signal_handler(fd, handler)                       [408-498]
  |
  +-- read signalfd event
  |
  +-- SIGCHLD:
  |     waitid(P_PID, handler->pid, WEXITED|WNOWAIT|WNOHANG)
  |                                              [423-427]
  |     if container init died:
  |         mark init_died = true
  |         check exit status
  |         close main loop                            [485-497]
  |
  +-- SIGHUP:
  |     restart container if configured
  |
  +-- other signals:
        forward to container init
```

### 6.2 容器退出检测

```
waitid(P_PID, handler->pid, &info, WEXITED|WNOWAIT|WNOHANG)
  +-- WIFEXITED(status)  -> normal exit
  +-- WIFSIGNALED(status) -> killed by signal
  +-- trigger main-loop exit -> lxc_poll() return
```

## 8. resolve_clone_flags()

**位置**：`start.c` 中的 clone 标志解析

根据容器配置生成 clone 系统调用的标志：

| 配置项 | 标志 | 含义 |
|--------|------|------|
| `lxc.namespace.clone` | 指定要创建的 | 白名单模式 |
| `lxc.namespace.keep` | 指定不创建的 | 黑名单模式 |
| 默认 | 全部 CLONE_NEW* | 最大隔离 |

标志分为三类：
- `ns_clone_flags`：通过 clone3() 创建
- `ns_unshare_flags`：子进程内通过 unshare() 创建
- `ns_on_clone_flags`：传给 clone() 的最终标志

部分命名空间需要在子进程内 unshare（如 CLONE_NEWCGROUP、CLONE_NEWTIME），
因为它们依赖子进程内的某些设置完成后才能创建。

## 9. ops 模式差异

```c
// start.c:2371-2404
static struct lxc_operations start_ops = {
    .start = start_cb,        // execvp(argv[0], argv)
    .post_start = post_start_cb,
};

// execute.c:21-53
static struct lxc_operations execute_start_ops = {
    .start = execute_start_cb, // lxc_container_init(argv)
    .post_start = NULL,
};
```

| 模式 | ops | 最终执行 |
|------|-----|----------|
| lxc-start | `start_ops` | `execvp("/sbin/init", ...)` |
| lxc-execute | `execute_start_ops` | `lxc_container_init()` → lxc-init 包装 |

两种模式共享完全相同的启动引擎（`__lxc_start`），仅在最终一步的 exec 行为不同。

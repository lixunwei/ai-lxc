# LXC 核心生命周期深度分析：启动引擎

## 1. 概述

LXC 的容器启动引擎位于 `src/lxc/start.c`，是整个系统最核心的代码。
启动过程通过三层函数实现：

```
__lxc_start()          顶层入口：初始化 → 生成 → 监控 → 清理
  ├── lxc_init()       预生成准备：信号、cgroup、钩子、安全
  ├── lxc_spawn()      核心生成：clone3() + 父子同步协议
  └── lxc_poll()       监控主循环：信号、命令、控制台
```

## 4. lxc_spawn() — 核心生成

**位置**：`start.c:1834-2170`

两种路径：

#### 继承命名空间路径

```

```

正常启动路径：

```

  ├── 回退 CLONE_ARGS_SIZE_VER0                   [1917-1921]
```

## 6. lxc_poll() — 监控主循环

**位置**：`start.c:763-847`

容器启动后，父进程（monitor）进入主循环：

```
lxc_poll(handler, post_start_fn)
  │                                              [763-847]
  ├── 创建主循环
  │     lxc_mainloop_open(descr)                   [772-778]
  │
  ├── 注册处理器：
  │     ├── 信号处理器（SIGCHLD 等）               [786-793]
  │     ├── seccomp 代理（如果启用）               [795-799]
  │     ├── 控制台 I/O 处理器                      [801-807]
  │     └── 命令套接字处理器                       [809-813]
  │
  ├── 运行主循环
  │     lxc_mainloop(descr, -1)                    [817-821]
  │     → epoll/io_uring 等待事件
  │     → 分发到注册的处理器
  │
  └── 循环结束（容器退出后）
        关闭所有 fd                                [822-846]
```

### 6.1 信号处理

```
signal_handler(fd, handler)                       [408-498]
  │
  ├── 读取 signalfd 事件
  │
  ├── SIGCHLD：
  │     waitid(P_PID, handler->pid, WEXITED|WNOWAIT|WNOHANG)
  │                                              [423-427]
  │     if 容器 init 已死亡:
  │         标记 init_died = true
  │         检查退出状态
  │         关闭主循环                            [485-497]
  │
  ├── SIGHUP：
  │     如果配置了重启，重启容器
  │
  └── 其他信号：
        转发给容器 init 进程
```

### 6.2 容器退出检测

```
waitid(P_PID, handler->pid, &info, WEXITED|WNOWAIT|WNOHANG)
  ├── WIFEXITED(status)  → 正常退出
  ├── WIFSIGNALED(status) → 被信号杀死
  └── 触发主循环退出 → lxc_poll() 返回
```

## 8. resolve_clone_flags()

**位置**：`start.c` 中的 clone 标志解析

根据容器配置生成 clone 系统调用的标志：

| 配置项 | 标志 | 含义 |
|--------|------|------|
| `lxc.namespace.clone` | 指定要创建的 | 白名单模式 |
| `lxc.namespace.keep` | 指定不创建的 | 黑名单模式 |
| 默认 | 全部 CLONE_NEW* | 最大隔离 |

标志分为三类：
- `ns_clone_flags`：通过 clone3() 创建
- `ns_unshare_flags`：子进程内通过 unshare() 创建
- `ns_on_clone_flags`：传给 clone() 的最终标志

部分命名空间需要在子进程内 unshare（如 CLONE_NEWCGROUP、CLONE_NEWTIME），
因为它们依赖子进程内的某些设置完成后才能创建。

## 9. ops 模式差异

```c
// start.c:2371-2404
static struct lxc_operations start_ops = {
    .start = start_cb,        // execvp(argv[0], argv)
    .post_start = post_start_cb,
};

// execute.c:21-53
static struct lxc_operations execute_start_ops = {
    .start = execute_start_cb, // lxc_container_init(argv)
    .post_start = NULL,
};
```

| 模式 | ops | 最终执行 |
|------|-----|----------|
| lxc-start | `start_ops` | `execvp("/sbin/init", ...)` |
| lxc-execute | `execute_start_ops` | `lxc_container_init()` → lxc-init 包装 |

两种模式共享完全相同的启动引擎（`__lxc_start`），仅在最终一步的 exec 行为不同。

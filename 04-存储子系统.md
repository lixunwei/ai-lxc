# 04 - 存储子系统

## 1. 存储抽象层（storage/storage.h/c）

### 1.1 统一后端接口

`struct lxc_storage_ops`（`storage.h:48-69`）定义了所有存储后端必须实现的操作：

| 操作 | 说明 |
|------|------|
| `detect()` | 自动检测路径是否属于此后端 |
| `mount()` | 挂载存储 |
| `umount()` | 卸载存储 |
| `destroy()` | 销毁存储 |
| `create()` | 创建新存储 |
| `clone_paths()` | 克隆路径准备 |
| `copy()` | 复制数据 |
| `snapshot()` | 创建快照 |
| `can_snapshot()` | 是否支持快照 |
| `can_backup()` | 是否支持备份 |

### 1.2 统一存储对象

`struct lxc_storage`（`storage.h:76-89`）：

```c
struct lxc_storage {
    const struct lxc_storage_ops *ops;  // 后端操作函数
    char *type;          // 后端类型名
    char *src;           // 源路径
    char *dest;          // 目标挂载点
    char *mntopts;       // 挂载选项
    int lofd;            // loop 设备 fd
    int nbd_idx;         // nbd 索引
    void *rootfs;        // rootfs 指针
};
```

### 1.3 注册的后端

`storage.c:170-180` 注册了以下后端（按优先级排列）：

| 后端 | 说明 | 快照支持 |
|------|------|----------|
| `dir` | 目录（默认） | ✗ |
| `zfs` | ZFS 数据集 | ✓ |
| `lvm` | LVM 逻辑卷 | ✓ |
| `rbd` | Ceph RBD 块设备 | ✓ |
| `btrfs` | Btrfs 子卷 | ✓ |
| `overlay` / `overlayfs` | OverlayFS 分层 | ✓ |
| `loop` | 循环设备（文件模拟块设备） | ✗ |
| `nbd` | 网络块设备 | ✗ |

## 2. 存储操作调度

### 2.1 自动识别

```
storage_query()                    ← storage.c:213-233
  ├── 优先按配置中的 __bdev_type 匹配
  └── 否则依次调用各后端 detect() 自动识别
```

### 2.2 按类型获取

```
storage_get()                      ← storage.c:235-256
  └── 按类型名字符串查找并构造 lxc_storage 对象
```

### 2.3 创建策略

```
storage_create()                   ← storage.c:544-584
  ├── 默认: "dir"
  ├── "best": 按优先级尝试 btrfs → zfs → lvm → dir → rbd
  └── 逗号分隔: 如 "lvm,dir" → 依次尝试直到成功
```

### 2.4 复制/克隆策略

```
storage_copy()                     ← storage.c:296-533
  ├── 根据原/新后端类型决定操作方式：
  │   ├── 同类型且支持快照 → snapshot
  │   ├── overlay → 创建分层
  │   ├── btrfs → subvolume snapshot
  │   ├── lvm → lvcreate -s
  │   ├── zfs → zfs clone
  │   └── 其他 → rsync 复制
  └── 处理后端切换的特殊路径
```

## 3. 各后端实现详解

### 3.1 目录后端（dir.c）

最简单的后端，直接使用目录作为 rootfs。

| 操作 | 实现 | 位置 |
|------|------|------|
| detect | 路径前缀 `dir:` 或真实目录 | `dir.c:109-125` |
| create | `mkdir -p`，src 设为 `dir:<path>` | `dir.c:57-93` |
| mount | 优先新 mount API / bind mount；否则 `mount("bind")` + 可选 remount ro | `dir.c:127-198` |
| umount | `umount2(MNT_DETACH)` | `dir.c:201-210` |
| clone | 不支持快照，仅路径替换 | `dir.c:23-55` |

### 3.2 Btrfs 后端（btrfs.c）

利用 Btrfs 的子卷和快照能力。

| 操作 | 实现 | 位置 |
|------|------|------|
| detect | `btrfs:` 前缀或检测真实 btrfs subvolume | `btrfs.c:169-189` |
| mount | bind mount (`MS_BIND\|MS_REC`) | `btrfs.c:191-214` |
| snapshot | `btrfs subvolume snapshot` | 由 storage.c 调度 |
| copy | `btrfs subvolume snapshot` 或 rsync | `storage.c:442-454` |

### 3.3 LVM 后端（lvm.c）

使用 LVM 逻辑卷作为 rootfs。

| 操作 | 实现 | 位置 |
|------|------|------|
| detect | `lvm:` 前缀或块设备 + `/sys/dev/block/.../dm/uuid` 以 `LVM-` 开头 | `lvm.c:175-212` |
| create | 调用 `lvcreate` 命令 | `lvm.c:40-100` |
| snapshot | 调用 `lvcreate -s` | `lvm.c:100-170` |
| mount | `mount_unknown_fs(src, dest, mntopts)` — 自动检测文件系统 | `lvm.c:214-230` |
| umount | 普通 `umount` | `lvm.c:232-240` |
| destroy | 调用 `lvremove` | — |

### 3.4 Overlay 后端（overlay.c）

主要用于**快照场景**，利用 OverlayFS 的分层机制。

```
overlay clonepaths()               ← overlay.c:35-47
  └── 要求 snap=true（仅用于快照）

目录结构：
  <container_path>/
    ├── delta/    ← upper layer（可写层）
    ├── work/     ← overlay workdir
    └── rootfs    ← 挂载点

src 格式：overlay:<lower>:<upper>
```

| 操作 | 实现 | 位置 |
|------|------|------|
| clone | 创建 delta/work 目录，构造 overlay 路径 | `overlay.c:65-119` |
| mount | `mount -t overlay -o lowerdir=...,upperdir=...,workdir=...` | — |
| snapshot | 复制 delta 层或创建新 overlay | `overlay.c:120-237` |

### 3.5 ZFS 后端（zfs.c）

使用 ZFS 数据集。

| 操作 | 实现 | 位置 |
|------|------|------|
| detect | `zfs:` 前缀；或 legacy 路径下通过 `zfs list/get` 判断 | `zfs.c:108-155` |
| mount | `mount(src, dest, "zfs", ...)` + `zfsutil,mntpoint=...` | `zfs.c:157-237` |
| snapshot | `zfs snapshot` + `zfs clone` | `zfs.c:39-75` |
| destroy | `zfs destroy` | — |

### 3.6 Loop 后端（loop.c）

将文件作为块设备使用。

| 操作 | 实现 | 位置 |
|------|------|------|
| detect | `loop:` 前缀或普通文件 | `loop.c:196-212` |
| create | 创建 rootdev 文件 → 格式化（默认 ext4, 1GiB） | `loop.c:113-184` |
| mount | 绑定到 loop 设备 → mount | — |
| clone | 不支持快照 | `loop.c:31-110` |
| destroy | 删除后端文件 | `loop.c:186-194` |

## 4. 挂载工具（mount_utils.c/h）

### 4.1 设计目标

统一处理新旧 mount API 的兼容性，支持：
- 旧式 `mount(2)` 系统调用
- 新 mount API（`fsopen/fsconfig/fsmount/move_mount`）
- Bind mount / idmapped mount / detached mount

### 4.2 新 Mount API 流程

```
fs_prepare()
  └── fsopen(fstype)              ← 打开文件系统上下文

fs_attach()
  ├── fsconfig(...)               ← 配置文件系统参数
  ├── fsmount(...)                ← 创建挂载
  └── move_mount(...)             ← 移动到目标挂载点
```

实现位于 `mount_utils.c:143-268`。

### 4.3 关键函数

| 函数 | 功能 | 位置 |
|------|------|------|
| `mnt_attributes_new()` | `MS_*` → `MOUNT_ATTR_*` 转换 | `mount_utils.c:47-90` |
| `mnt_attributes_old()` | `MOUNT_ATTR_*` → `MS_*` 转换 | `mount_utils.c:92-141` |
| `create_detached_idmapped_mount()` | `open_tree` + `MOUNT_ATTR_IDMAP` | `mount_utils.c:270-302` |
| `__fd_bind_mount()` | detached tree + `mount_setattr` + `move_mount` | `mount_utils.c:332-387` |
| `add_required_remount_flags()` | 根据 `statvfs` 补齐 `MS_NOSUID/MS_NODEV/MS_RDONLY/...` | `mount_utils.c:489-536` |
| `can_use_mount_api()` | 探测内核是否支持新 mount API | `mount_utils.c:538-575` |
| `can_use_bind_mounts()` | 探测 bind-mount 扩展支持 | `mount_utils.c:577-610` |
| `mount_at()` | 最终回退到传统 `mount(2)` | `mount_utils.c:613+` |

### 4.4 在存储后端中的应用

`dir.c` 已优先使用新 mount API：

```
dir_mount()
  ├── [优先] fd_bind_mount() / move_detached_mount()  ← 新 API
  └── [回退] mount(..., "bind", MS_BIND|MS_REC)       ← 旧 API
```

这种设计确保了在新旧内核上都能正常工作。

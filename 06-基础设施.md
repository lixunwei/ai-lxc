# 06 - 基础设施

## 1. 事件循环（mainloop.c/h）

### 1.1 双后端设计

LXC 的事件循环同时支持 **epoll** 和 **io_uring** 两种后端：

```
lxc_mainloop_open()                <- mainloop.c:507-519
  +-[prefer] io_uring (needs liburing)
  `-[fallback] epoll
```

### 1.2 Handler 结构

每个事件处理器记录（`mainloop.h:24-64`）：

```c
struct mainloop_handler {
    int fd;                    // 监听的文件描述符
    lxc_mainloop_callback_t callback;  // 回调函数
    lxc_mainloop_cleanup_t cleanup;    // 清理函数
    bool oneshot;              // 是否一次性
    char *name;                // 调试名称
};
```

### 1.3 回调返回值

回调函数的返回值决定后续行为：

| 返回值 | 行为 |
|--------|------|
| `LXC_MAINLOOP_CONTINUE` | 继续监听 |
| `LXC_MAINLOOP_DISARM` | 移除此 handler |
| `LXC_MAINLOOP_CLOSE` | 关闭整个事件循环 |
| `LXC_MAINLOOP_ERROR` | 错误，关闭循环 |

### 1.4 io_uring 支持

io_uring 路径支持 oneshot/multishot poll：
- 完成后根据 `IORING_CQE_F_MORE` 决定是否重新注册
- 实现位于 `mainloop.c:235-331`

### 1.5 epoll 路径

更简单的实现：
```
epoll_wait() -> match handler -> callback -> handle by return value
```
实现位于 `mainloop.c:334-375`。

## 2. 日志系统（log.c/h）

### 2.1 设计模式：事件 + Appender 链

```
Log event
  `- appender chain:
      +- stderr appender    <- error and above
      +- logfile appender   <- file log
      +- syslog appender    <- system log
      `- dlog appender      <- other
```

实现位于 `log.c:137-186`。

### 2.2 容器日志分离

日志可按容器独立输出（`log.c:93-105`）：

```
Priority:
  1. current_config->logfd   <- container log fd
  2. lxc_log_fd              <- global log fd
```

### 2.3 日志能力

- **stderr 输出**：error 及以上级别
- **syslog 集成**：标准 syslog 协议
- **文件日志**：支持轮转、截断、大小限制
- **容器名注入**：日志中自动包含容器名/前缀

### 2.4 使用方式

每个源文件通过宏绑定模块名：

```c
lxc_log_define(network, lxc);   // 声明 "network" 模块属于 "lxc" 分类
```

## 3. 锁机制（lxclock.c/h）

### 3.1 两种锁类型

**文件锁**（用于容器间互斥）：

```
Lock file path: <rundir>/lxc/lock/<lxcpath>/.<name>

Lock strategy:
  +-[prefer] F_OFD_SETLKW  <- Open File Description Locks (thread-safe)
  `-[fallback] flock()     <- BSD file lock
```

实现位于 `lxclock.c:79-117, 159-207`。

**匿名信号量**（用于进程内同步）：

```
sem_init()
sem_wait() / sem_timedwait()
sem_post()
```

实现位于 `lxclock.c:119-157, 209-228`。

### 3.2 创建逻辑

```
lxc_newlock()                      <- lxclock.c:135-157
  +- name given -> file lock (container-level mutex)
  `- no name    -> anonymous semaphore (in-process sync)
```

## 4. IPC 命令通道（commands.c/h）

### 4.1 设计概述

`commands.c` 是 LXC 工具与运行中容器 monitor 进程之间的 IPC 层（`commands.c:36-59`）：

- **协议**：事务式请求/响应
- **传输**：Unix domain socket
- **附加数据**：SCM_CREDENTIALS（进程凭证）+ SCM_RIGHTS（文件描述符传递）

### 4.2 支持的命令

| 命令 | 功能 |
|------|------|
| `GET_TTY` | 获取 tty 文件描述符 |
| `STOP` | 终止容器 |
| `GET_STATE` | 获取容器状态 |
| `GET_INIT_PID` | 获取 init 进程 PID |
| `GET_INIT_PIDFD` | 获取 init 进程 pidfd |
| `GET_CGROUP_FD` | 获取 cgroup 文件描述符 |
| `GET_SECCOMP_NOTIFIER_FD` | 获取 seccomp notifier fd |
| `FREEZE` / `UNFREEZE` | 冻结/解冻容器 |
| ... | 其他 |

实现位于 `commands.c:63-93`。

### 4.3 文件描述符传递

响应中可能包含文件描述符，通过 `SCM_RIGHTS` 传递：

```
lxc_cmd_rsp_recv_fds()             <- commands.c:120-149, 168-239
  `- max fd count depends on command type
```

### 4.4 协议兼容性

> **重要设计原则**：协议语义不可随意变更，否则会破坏新旧工具与运行中容器监控进程的兼容性。
>
> — `commands.c:50-58`

## 6. 同步机制（sync.c/h）

### 6.1 用途

`sync.c` 提供父子进程间的同步原语，用于容器启动过程中的多阶段握手。

### 6.2 同步消息

通过 socket pair 发送整数消息：

```c
// 典型同步点
START_SYNC_STARTUP          // 子进程已启动
START_SYNC_POST_CONFIGURE   // 配置已完成
START_SYNC_CGROUP_LIMITS    // 需要设置 cgroup limits
START_SYNC_READY_START      // 准备好启动
```

### 6.3 工作方式

```
Parent                          Child
   |                              |
   |<-- SYNC_STARTUP ------------|  child reports started
   |                              |
   +-- [run config ops]           |
   |                              |
   +-- SYNC_POST_CONFIGURE ------>|  notify config done
   |                              |
   |<-- SYNC_CGROUP_LIMITS ------|  child requests cgroup limits
   |                              |
   +-- [set cgroup limits]        |
   |                              |
   +-- SYNC_READY_START --------->|  notify resume
   |                              |
   |                              +-- [exec user command]
   |                              |
```

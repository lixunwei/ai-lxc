# LXC 基础设施深度分析（一）：事件循环

## 1. 概述

LXC 的事件循环是容器监控进程（monitor）的核心，负责处理信号、命令通道、终端 I/O、seccomp notify 等异步事件。实现采用 **双后端架构**：优先使用 **io_uring**，不支持时回退到 **epoll**。

核心文件：
- `src/lxc/mainloop.h`（~65 行）— 接口定义
- `src/lxc/mainloop.c`（~545 行）— 双后端实现

lxc_mainloop_open(descr) {

    ret = __io_uring_open(descr);

## 5. 事件循环主体

### 5.1 分发入口

`mainloop.c:378-384`：

```c
lxc_mainloop(descr, timeout_ms) {
    if (descr->type == LXC_MAINLOOP_IO_URING)
        return __lxc_mainloop_io_uring(descr, timeout_ms);
    else
        return __lxc_mainloop_epoll(descr, timeout_ms);
}
```

### 5.2 io_uring 事件循环

`mainloop.c:235-331`，`__lxc_mainloop_io_uring()`：

```
__lxc_mainloop_io_uring(descr, timeout_ms)
  loop:
    // 等待完成事件（CQE）
    if (timeout_ms >= 0):
        io_uring_wait_cqe_timeout(ring, &cqe, &timeout)
    else:
        io_uring_wait_cqe(ring, &cqe)

    // 处理取消确认
    if (cqe 是取消结果):
        标记 CANCEL_RECEIVED / CANCEL_SUCCESS
        goto next_cqe

    // 获取关联的 handler
    handler = io_uring_cqe_get_data(cqe)

    // 调用回调
    ret = handler->callback(handler->fd, handler->data, descr, cqe->res)

    switch (ret):
        case CONTINUE:
            // 重新提交 poll SQE
            __io_uring_arm(descr, handler)
        case DISARM:
            // 从链表移除，不再监听
            list_del(handler)
            cleanup(handler)
        case CLOSE:
            // 退出整个事件循环
            return
        case ERROR:
            // 错误退出
            return -1
```

### 5.3 epoll 事件循环

`mainloop.c:334-376`，`__lxc_mainloop_epoll()`：

```
__lxc_mainloop_epoll(descr, timeout_ms)
  loop:
    nfds = epoll_wait(epfd, events, MAX_EVENTS, timeout_ms)

    for i = 0 to nfds:
        handler = events[i].data.ptr

        ret = handler->callback(handler->fd, handler->data, descr, events[i].events)

        switch (ret):
            case CONTINUE: continue
            case DISARM:   del_handler + cleanup
            case CLOSE:    return 0
            case ERROR:    return -1
```

## 6. 信号处理

### 6.1 signalfd 创建

`start.c:376-406`，`setup_signal_fd()`：

```c
setup_signal_fd(old_mask) {
    sigset_t mask;
    sigfillset(&mask);

    // 阻塞所有信号（后续通过 signalfd 处理）
    sigprocmask(SIG_BLOCK, &mask, old_mask);

    // 创建 signalfd
    return signalfd(-1, &mask, SFD_CLOEXEC);
}
```

### 6.2 信号回调

`start.c:408-498`，`signal_handler()`：

```
signal_handler(fd, data, descr, events)
  // 读取 signalfd_siginfo
  read(fd, &siginfo, sizeof(siginfo))

  switch (siginfo.ssi_signo):
      case SIGCHLD:
          // 使用 waitid() 检查子进程状态
          waitid(P_PID, init_pid, &info, WNOHANG|WEXITED)
          if (子进程已退出):
              return LXC_MAINLOOP_CLOSE  // 退出事件循环

      case SIGTERM / SIGINT:
          // 转发信号给容器 init 进程
          kill(init_pid, siginfo.ssi_signo)
          return LXC_MAINLOOP_CONTINUE

      default:
          return LXC_MAINLOOP_CONTINUE
```

## 7. 容器生命周期中的使用

`start.c` 中的事件循环使用（`772-845`）：

```
lxc_poll(handler)

  ═══ 初始化 ═══
  1. lxc_mainloop_open(&descr)                     // 创建主事件循环
  2. lxc_mainloop_open(&descr_console)              // 创建终端事件循环（可选）

  ═══ 注册 Handler ═══
  3. lxc_mainloop_add_handler(signal_fd, signal_handler)       // 信号
  4. lxc_seccomp_setup_proxy(handler, &descr)                  // seccomp notify
  5. lxc_terminal_mainloop_add(&descr)                         // 终端 I/O
  6. lxc_cmd_mainloop_add(name, &descr, handler)               // 命令通道

  ═══ 运行 ═══
  7. lxc_mainloop(&descr, -1)                      // 无限等待，直到容器退出

  ═══ 终端后续处理 ═══
  8. lxc_mainloop(&descr_console, 0)               // 刷新剩余终端输出

  ═══ 清理 ═══
  9. lxc_mainloop_close(&descr_console)
  10. lxc_mainloop_close(&descr)
  11. close(signal_fd)
```

### 事件循环中监听的 fd

| fd 类型 | 来源 | 回调 | 作用 |
|---------|------|------|------|
| signalfd | `setup_signal_fd()` | `signal_handler()` | 处理 SIGCHLD/SIGTERM 等 |
| 命令 socket | `lxc_server_init()` | `lxc_cmd_accept()` | 接受 IPC 命令连接 |
| 终端 PTY fd | `lxc_terminal_setup()` | 终端 I/O 回调 | 转发容器终端输入输出 |
| seccomp notify fd | seccomp 加载后 | `seccomp_notify_handler()` | 拦截系统调用 |

## 8. 设计总结

1. **双后端透明切换**：io_uring 和 epoll 通过统一的 handler 注册/回调接口隐藏差异，上层代码无需关心底层实现。

2. **io_uring 优势**：减少系统调用次数（批量提交/完成），内核轮询模式（SQPOLL）可进一步降低延迟。

3. **回调驱动**：所有异步事件通过统一的回调函数处理，返回码控制后续行为（继续/解除/关闭/错误）。

4. **信号统一化**：通过 signalfd 将信号转换为文件描述符事件，避免异步信号处理的复杂性和竞争条件。

5. **超时支持**：事件循环接受 timeout 参数，`-1` 表示无限等待，`0` 用于非阻塞刷新（如终端后续处理）。

# 05 - 安全子系统

## 1. Linux 安全模块（LSM）

### 1.1 抽象层

`struct lsm_ops`（`lsm/lsm.h:16-37`）提供统一接口：

| 接口 | 功能 |
|------|------|
| `enabled()` | 是否启用 |
| `process_label_get()` | 获取进程安全标签 |
| `process_label_set()` | 设置进程安全标签 |
| `keyring_label_set()` | 设置 keyring 安全标签 |
| `prepare()` | 启动前准备（如加载 profile） |
| `cleanup()` | 停止后清理 |
| `*_at()` | fd-based 变体 |

### 1.2 LSM 选择逻辑

`lsm_init_static()`（`lsm/lsm.c:22-39`）：

```
检测顺序：
  1. AppArmor → 如果启用，使用 AppArmor
  2. SELinux  → 如果启用，使用 SELinux
  3. nop      → 否则使用空实现
```

### 1.3 SELinux 实现（lsm/selinux.c）

直接使用 libselinux API（`selinux.c:34-185`）：

| 操作 | 实现 |
|------|------|
| 获取标签 | `getpidcon_raw()` |
| 设置执行标签 | `setexeccon_raw()` |
| 设置当前标签 | `setcon_raw()` |
| 设置 keyring 标签 | `setkeycreatecon_raw()` |
| fd-based 读写 | `/proc/<pid>/attr/current\|exec` |

### 1.4 AppArmor 实现（lsm/apparmor.c）

AppArmor 支持更为复杂（`apparmor.c:423-503, 980-1321`）：

- **Profile 管理**：自己拼接 profile、namespace、缓存目录，调用 `apparmor_parser` 加载/卸载
- **标签读写**：通过 `/proc/<pid>/attr/apparmor/current`（新接口）或 `/proc/<pid>/attr/current`（旧接口）
- **标签设置**：写入 `changeprofile <profile>` 字符串

### 1.5 Nop 实现（lsm/nop.c）

所有接口返回空/成功，用于不需要 LSM 的环境（`nop.c:9-77`）。

## 2. Seccomp 过滤（seccomp.c, lxcseccomp.h）

### 2.1 数据结构

`struct lxc_seccomp`（`lxcseccomp.h:67-92`）：

```c
struct lxc_seccomp {
    char *seccomp;           // 配置文件路径
    scmp_filter_ctx seccomp_ctx;  // libseccomp 过滤器上下文
    bool allow_nesting;      // 嵌套控制
    // seccomp notify 相关字段
    int notifier_fd;
    // ...
};
```

### 2.2 配置格式

支持两种配置格式：

**v1 格式**（`seccomp.c:45-71`）：
```
# 每行一个 syscall 号，全部转为 ALLOW
1
2
3
```

**v2 格式**（`seccomp.c:95-295, 623-840`）：
```
2                           # 版本号
allowlist                   # 策略：allowlist 或 denylist
[x86_64]                    # 架构分组
read allow                  # syscall 动作
write allow
ioctl errno 1               # 带参数
mmap allow [0,0,PROT_EXEC]  # 参数比较表达式
[x86]                       # 另一个架构
...
```

### 2.3 过滤器构建流程

```
seccomp_init()                      ← 创建上下文
  ├── 关闭/调整 NNP（No New Privileges）
  ├── 可选 TSKIP
  │
  ├── 按架构分别构建 ctx：
  │   ├── seccomp_syscall_resolve_name_arch()  ← 解析系统调用名
  │   └── seccomp_rule_add_exact_array()       ← 添加规则
  │
  └── 合并多架构 ctx
```

实现位于 `seccomp.c:358-495, 504-586`。

### 2.4 Seccomp Notify

在 `lxc-attach` 中，seccomp notify fd 通过 IPC 传回父进程做监督（`attach.c:1358-1369, 1773-1781`）：

```
子进程：加载 seccomp → 获得 notify fd → 通过 IPC 传给父进程
父进程：监听 notify fd → 对被拦截的系统调用做裁决
```

## 3. 容器附着（attach.c/h）

### 3.1 设计概述

`lxc-attach` 允许在运行中的容器内执行命令。核心设计是**两次 fork**。

### 3.2 整体流程

```
lxc_attach()                        ← attach.c:1437-1501
  │
  ├── 1. 读取容器上下文
  │     ├── seccomp 配置
  │     ├── no_new_privs 标志
  │     ├── namespace fd / pidfd
  │     └── 终端设置
  │
  ├── 2. 第一次 fork → "过渡进程"
  │     │                           attach.c:1542-1600
  │     │
  │     └── 3. setns() 进入容器 namespaces
  │           │
  │           ├── [新路径] setns(init_pidfd, ns_flags)
  │           │                     attach.c:634-650
  │           │
  │           └── [老路径] 逐个打开 /proc/<init>/ns/* → setns(fd, flag)
  │                                 attach.c:573-700
  │
  ├── 4. 第二次 fork → clone(CLONE_PARENT) → "附着进程"
  │     │                           attach.c:1600-1662
  │     │
  │     └── 5. 在附着进程中：
  │           ├── 设置 uid/gid/groups
  │           ├── 设置 LSM 标签
  │           ├── 加载 seccomp
  │           ├── 设置 PR_SET_NO_NEW_PRIVS
  │           ├── 设置 stdin/stdout/stderr
  │           └── exec(用户命令)
  │                                 attach.c:1240-1377
  │
  └── 6. 父进程：
        ├── 接收 LSM label fd、seccomp notifier fd（通过 IPC）
        └── 如果终端模式：mainloop 处理 stdin ↔ pty 转发
                                    attach.c:1385-1415, 1716-1803
```

### 3.3 两次 fork 的原因

使用 `CLONE_PARENT` 创建附着进程，使其成为过渡进程的**兄弟**而非子进程。这样：

1. **PID namespace 语义正确**：附着进程的 PID 在容器 PID namespace 中是正确的
2. **User namespace 语义正确**：权限检查在正确的上下文中进行
3. **过渡进程可以安全退出**：不影响附着进程的生命周期

## 4. CRIU 检查点/恢复（criu.c/h）

### 4.1 设计理念

LXC **不自己实现** checkpoint/restore，而是负责：
- 准备上下文
- 生成 CRIU 命令参数
- 执行 CRIU
- 处理容器侧的资源/namespace/console 绑定

### 4.2 核心操作

| 操作 | 说明 | 位置 |
|------|------|------|
| dump | 拼接 `criu dump` 命令行 | `criu.c:159-200` |
| pre-dump | 增量预转储 | `criu.c:200-220` |
| restore | 拼接 `criu restore` 命令行 | `criu.c:218-260` |

### 4.3 dump 前处理

```
dump 前：
  ├── 让 cgroup ops "escape" 到根 cgroup
  │   └── 避免 lxcfs / cgroup 访问问题
  │       criu.c:173-182
  │
  └── 传递参数：
      ├── cgroup 路径
      ├── 网络外部对象映射
      ├── LSM 相关参数
      └── TTY 配置
```

### 4.4 restore 处理

恢复时需要重建：
- TTY 终端
- Cgroup 层级
- 网络外部对象映射
- LSM/容器相关参数

命令参数按 CRIU 版本差异动态适配（通过 `criu --version` 和 `criu check --feature` 检测）。

## 5. ID 映射（idmap_utils.c/h）

### 5.1 写入映射

`write_id_mapping()`（`idmap_utils.c:23-65`）：

```
直接写 /proc/<pid>/uid_map 或 /proc/<pid>/gid_map

特殊处理：
  非 root 写 gid_map 时，先向 /proc/<pid>/setgroups 写 "deny\n"
```

### 5.2 映射策略

`lxc_map_ids()`（`idmap_utils.c:129-257`）支持两条路径：

```
lxc_map_ids()
  ├── 检查是否映射 host root
  │
  ├── [路径 A] 借助 shadow utils（优先）
  │   ├── 检查 newuidmap/newgidmap 是否存在且有权限
  │   └── 执行 newuidmap/newgidmap 命令
  │
  └── [路径 B] 直接写 proc 文件
      └── write_id_mapping()
```

自动拼装多段 mapping 字符串，支持复杂的 UID/GID 映射配置。

### 5.3 在容器启动中的应用

启动时，父进程在 clone 之后、子进程继续执行之前，写入 UID/GID 映射：

```
父进程                              子进程
  │                                   │
  ├── clone() ───────────────────────►│
  │                                   ├── 等待同步信号
  │                                   │
  ├── lxc_map_ids()                   │
  │   ├── 写 uid_map                  │
  │   └── 写 gid_map                  │
  │                                   │
  ├── 同步信号 ──────────────────────►│
  │                                   ├── 继续执行
  │                                   └── 此时 UID/GID 映射已生效
```

# 01 - 核心生命周期

## 1. 容器 API（lxccontainer.h/c）

### 1.1 核心数据结构

`struct lxc_container` 是用户与 LXC 交互的主要对象，定义于 `src/lxc/lxccontainer.h:50-520`。

**私有字段**：
- `name` — 容器名称
- `configfile` — 配置文件路径
- `pidfile` — PID 文件路径
- `slock` / `privlock` — 同步锁
- `lxc_conf` — 指向 `struct lxc_conf` 的指针（容器完整配置）
- `numthreads` — 引用计数

**公共 API 函数指针**（在初始化时绑定到具体实现）：

| 函数指针 | 功能 |
|----------|------|
| `create` / `createl` | 创建容器 |
| `start` / `startl` | 启动容器 |
| `stop` | 停止容器 |
| `destroy` | 销毁容器 |
| `shutdown` | 优雅关闭 |
| `wait` | 等待状态变更 |
| `get_config_item` / `set_config_item` | 读写配置项 |
| `clear_config` | 清空配置 |
| `attach` / `attach_run_wait` | 附着到运行中容器 |
| `snapshot` / `snapshot_list` / `snapshot_restore` | 快照管理 |
| `clone` | 克隆容器 |
| ... | 其他 |

### 1.2 生命周期主线

#### 创建（Create）

```
__lxcapi_create()
  ├── 创建容器目录结构
  ├── 创建 rootfs（选择存储后端）
  ├── 执行模板脚本
  ├── 保存配置到文件
  └── 失败时回滚删除
```

实现位于 `src/lxc/lxccontainer.c:1755-1920`。

#### 启动（Start）

```
lxcapi_start() / lxcapi_startl()
  └── do_lxcapi_start()
        └── __lxc_start()   ← 进入 start.c 的核心启动引擎
```

实现位于 `src/lxc/lxccontainer.c:1114-1159`。

#### 停止（Stop）

```
do_lxcapi_stop()
  └── 通过命令 socket 向 monitor 进程发送 stop 命令
```

实现位于 `src/lxc/lxccontainer.c:1162-1172`。

#### 销毁（Destroy）

```
do_lxcapi_destroy() / container_destroy()
  ├── 删除容器存储
  └── 删除容器目录
```

## 2. 容器启动引擎（start.h/c）

### 2.1 运行时核心对象

`struct lxc_handler` 是启动过程中的核心运行时对象，定义于 `src/lxc/start.h:21-137`：

```
struct lxc_handler {
    // Namespace 相关
    ns_clone_flags      // 需要 clone 的 namespace
    ns_on_clone_flags   // clone 时传递的 flags
    ns_unshare_flags    // clone 后 unshare 的 flags
    clone_flags         // 最终 clone flags

    // 进程 / 文件描述符
    pid                 // 容器 init 进程 PID
    pidfd               // pidfd
    sigfd               // signalfd
    data_sock[2]        // 数据传递 socket pair
    state_socket_pair[2]// 状态通知 socket pair
    sync_sock[2]        // 同步 socket pair

    // 配置
    conf                // 指向 lxc_conf
};
```

### 2.2 启动流程

完整的启动时序（`src/lxc/start.c:2198-2308`）：

```
__lxc_start()
  │
  ├── 1. lxc_init()
  │     ├── 初始化 handler
  │     ├── 创建状态 socket
  │     └── 设置信号处理
  │
  ├── 2. 处理块设备 / cgroup monitor
  │
  ├── 3. resolve_clone_flags()          ← 决定哪些 namespace 要 clone/unshare/inherit
  │     ├── 解析 ns_clone / ns_keep / ns_share 配置
  │     └── 修正 cgroup/time/user/net 的特殊顺序
  │
  ├── 4. lxc_inherit_namespaces()       ← 继承已有 namespace
  │
  ├── 5. lxc_rootfs_init()              ← 初始化根文件系统
  │
  ├── 6. lxc_spawn()                    ← 核心：fork/clone 出容器进程
  │     ├── 建立同步与 data socket
  │     ├── 创建 cgroup
  │     ├── [分支 A] 需要继承 namespace：
  │     │     ├── lxc_clone(do_share_ns, CLONE_VFORK|CLONE_VM|CLONE_FILES)
  │     │     ├── 中间进程 setns() 继承已有 namespace
  │     │     └── clone() 出容器 init
  │     ├── [分支 B] 直接创建：
  │     │     └── clone3()，可选 CLONE_INTO_CGROUP
  │     └── 父子同步（START_SYNC_* 多阶段握手）
  │
  └── 7. lxc_poll()                     ← 进入事件循环，监控容器
```

### 2.3 Namespace 设置要点

- `CLONE_NEWNET` 在 `CLONE_NEWUSER` **之后** unshare，确保非特权场景下权限正确
  （`start.c:1288-1298`）
- `CLONE_NEWCGROUP` / `CLONE_NEWTIME` 可能先 clone 后 unshare
  （`start.c:1371-1445`）
- 父子进程通过 `START_SYNC_*` 消息进行多阶段同步握手
  （`start.c:1364-1547, 2009-2140`）

### 2.4 父子同步时序图

```
       父进程（monitor）                    子进程（容器 init）
           │                                      │
           ├── clone3() / clone() ────────────────►│
           │                                      │
           │◄── START_SYNC_STARTUP ───────────────│  (子进程已启动)
           │                                      │
           ├── 设置 UID/GID mapping ──────────────►│
           ├── 设置 cgroup ───────────────────────►│
           │                                      │
           ├── START_SYNC_POST_CONFIGURE ─────────►│
           │                                      │  (子进程设置 namespace)
           │◄── START_SYNC_CGROUP_LIMITS ─────────│
           │                                      │
           ├── 设置 cgroup limits ────────────────►│
           ├── START_SYNC_READY_START ────────────►│
           │                                      │  (子进程 exec)
           └── 进入 lxc_poll() 事件循环            │
```

## 3. 配置管理（conf.h/c, confile.h/c）

### 3.1 核心配置结构

`struct lxc_conf` 是容器的完整配置，定义于 `src/lxc/conf.h:411-586`，包含极其丰富的字段：

| 字段分类 | 说明 |
|----------|------|
| `id_map` | UID/GID 映射列表 |
| `netdevs` | 网络设备列表 |
| `rootfs` | 根文件系统配置 (`struct lxc_rootfs`) |
| `mount_entries` | 挂载点列表 |
| `caps` / `keepcaps` | 能力白名单/黑名单 |
| `tty_info` / `console` | TTY 和控制台配置 |

| `environment` | 环境变量 |
| `seccomp` | Seccomp 配置 |
| `cgroup` / `cgroup2` | Cgroup 控制参数 |
| `sysctl` / `proc` | sysctl 和 proc 参数 |
| `signals` | 信号配置 |
| `timens` | 时间命名空间偏移 |

#### 关键子结构

**`struct lxc_rootfs`**（`conf.h:242-259`）：
```c
struct lxc_rootfs {
    char *path;         // rootfs 路径
    char *mount;        // 挂载点
    struct lxc_storage *storage;  // 存储后端
    char *options;      // 挂载选项
    // ...

```

**`struct id_map`**（`conf.h:188-192`）：
```c
struct id_map {
    enum idtype idtype; // u_type (uid/gid)

    unsigned long nsid;
    unsigned long range;
};
```

### 3.2 配置解析机制

配置解析使用**跳转表**模式（`confile.c:191-260`）：

```c
// 每个 lxc.* 配置键对应三个回调
struct lxc_config_t {
    char *name;
    config_set_cb set;   // 设置值
    config_get_cb get;   // 获取值
    config_clr_cb clr;   // 清除值
};

// 全局跳转表
static struct lxc_config_t config_jump_table[] = {
    { "lxc.rootfs.path",    set_config_rootfs_path,    get_config_rootfs_path,    clr_config_rootfs_path },
    { "lxc.mount.entry",    set_config_mount_entry,    get_config_mount_entry,    clr_config_mount_entry },
    { "lxc.net",            set_config_net,            get_config_net,            clr_config_net },
    // ... 几十个配置项
};
```

### 3.3 配置生命周期

```
lxc_conf_init()       ← 分配并初始化配置对象
    │
    ├── 加载配置文件 → config_jump_table[].set() 回调
    │
    ├── append_unexp_config_line()  ← 保存原始文本（用于 clone/save）
    │
    ├── [运行时使用配置]
    │
    └── lxc_conf_free()  ← 释放所有配置
```

## 4. 状态管理（state.h/c）

### 4.1 状态枚举

容器可处于以下状态（`state.h:12-22`）：

```
STOPPED → STARTING → RUNNING → STOPPING → STOPPED
                       ↓         ↑
                    FREEZING → FROZEN → THAWED(→RUNNING)
                       ↓
                    ABORTING → STOPPED
```

| 状态 | 说明 |
|------|------|
| `STOPPED` | 已停止 |
| `STARTING` | 正在启动 |
| `RUNNING` | 运行中 |
| `STOPPING` | 正在停止 |
| `ABORTING` | 异常中止 |
| `FREEZING` | 正在冻结 |
| `FROZEN` | 已冻结 |
| `THAWED` | 已解冻（映射到 RUNNING） |

### 4.2 状态通知机制

```
lxc_set_state()                         ← start.c:586-604
  ├── 通过匿名 socket pair 通知
  ├── lxc_serve_state_clients()         ← 通知所有订阅的客户端
  │     ├── 更新 handler->state
  │     ├── THAWED 特判 → 映射为 RUNNING
  │     └── 向订阅客户端发送状态消息
  └── 兼容 legacy monitord 广播
```

### 4.3 辅助函数

- `lxc_state2str()` — 状态枚举转字符串（`state.c:29-42`）
- `lxc_str2state()` — 字符串转状态枚举（`state.c:44-53`）

## 5. 容器 Init 进程（cmd/lxc_init.c, initutils.c）

### 5.1 入口

`lxc_init.c` 做命令行解析后调用 `lxc_container_init()`（`cmd/lxc_init.c:98-168`）。

### 5.2 Init 核心逻辑

真正的 init 逻辑在 `initutils.c:482-710`：

```
lxc_container_init()
  │
  ├── 设置信号处理，屏蔽信号
  ├── 删除自身可执行文件引用
  │
  ├── fork() ─────────────────────► 子进程：执行用户指定的命令
  │                                  └── exec(cmd)
  │
  └── 父进程：成为容器内的 "init" 监督者
        ├── 改进程名为 "init"
        ├── 关闭继承的 FD（close_range / close_inherited）
        ├── 捕获 SIGTERM/SIGPWR/SIGALRM
        │     ├── SIGTERM → shutdown / kill child
        │     └── SIGPWR → 类似 power off
        └── wait() 循环
              └── 维持 PID 1 风格的 "回收孤儿进程"
```

这个设计确保了：
1. 容器内始终有 PID 1 进程负责回收僵尸进程
2. 信号处理语义与真实 init 一致
3. 用户命令在子进程中独立运行

# 01 - 核心生命周期

## 1. 容器 API（lxccontainer.h/c）

### 1.1 核心数据结构

`struct lxc_container` 是用户与 LXC 交互的主要对象，定义于 `src/lxc/lxccontainer.h:50-520`。

**私有字段**：
- `name` — 容器名称
- `configfile` — 配置文件路径
- `pidfile` — PID 文件路径
- `slock` / `privlock` — 同步锁
- `lxc_conf` — 指向 `struct lxc_conf` 的指针（容器完整配置）
- `numthreads` — 引用计数

**公共 API 函数指针**（在初始化时绑定到具体实现）：

| 函数指针 | 功能 |
|----------|------|
| `create` / `createl` | 创建容器 |
| `start` / `startl` | 启动容器 |
| `stop` | 停止容器 |
| `destroy` | 销毁容器 |
| `shutdown` | 优雅关闭 |
| `wait` | 等待状态变更 |
| `get_config_item` / `set_config_item` | 读写配置项 |
| `clear_config` | 清空配置 |
| `attach` / `attach_run_wait` | 附着到运行中容器 |
| `snapshot` / `snapshot_list` / `snapshot_restore` | 快照管理 |
| `clone` | 克隆容器 |
| ... | 其他 |

### 1.2 生命周期主线

## 2. 容器启动引擎（start.h/c）

### 2.1 运行时核心对象

`struct lxc_handler` 是启动过程中的核心运行时对象，定义于 `src/lxc/start.h:21-137`：

```
struct lxc_handler {
    // Namespace 相关
    ns_clone_flags      // 需要 clone 的 namespace
    ns_on_clone_flags   // clone 时传递的 flags
    ns_unshare_flags    // clone 后 unshare 的 flags
    clone_flags         // 最终 clone flags

    // 进程 / 文件描述符
    pid                 // 容器 init 进程 PID
    pidfd               // pidfd
    sigfd               // signalfd
    data_sock[2]        // 数据传递 socket pair
    state_socket_pair[2]// 状态通知 socket pair
    sync_sock[2]        // 同步 socket pair

    // 配置
    conf                // 指向 lxc_conf
};
```

### 2.2 启动流程

完整的启动时序（`src/lxc/start.c:2198-2308`）：

```
__lxc_start()
  |
  +-- 1. lxc_init()
  |     +-- init handler
  |     +-- create state socket
  |     +-- setup signal handling
  |
  +-- 2. handle block dev / cgroup monitor
  |
  +-- 3. resolve_clone_flags()          <- decide which ns to clone/unshare/inherit
  |     +-- parse ns_clone / ns_keep / ns_share
  |     +-- fix cgroup/time/user/net order
  |
  +-- 4. lxc_inherit_namespaces()       <- inherit existing ns
  |
  +-- 5. lxc_rootfs_init()              <- init rootfs
  |
  +-- 6. lxc_spawn()                    <- core: fork/clone container proc
  |     +-- create sync and data sockets
  |     +-- create cgroup
  |     +-- [path A] inherit ns:
  |     |     +-- lxc_clone(do_share_ns, CLONE_VFORK|CLONE_VM|CLONE_FILES)
  |     |     +-- middle proc setns()
  |     |     +-- clone() container init
  |     +-- [path B] direct create:
  |     |     +-- clone3(),optional CLONE_INTO_CGROUP
  |     +-- parent/child sync(START_SYNC_* multi-stage handshake)
  |
  +-- 7. lxc_poll()                     <- enter event loop, monitor container
```

### 2.3 Namespace 设置要点

- `CLONE_NEWNET` 在 `CLONE_NEWUSER` **之后** unshare，确保非特权场景下权限正确
  （`start.c:1288-1298`）
- `CLONE_NEWCGROUP` / `CLONE_NEWTIME` 可能先 clone 后 unshare
  （`start.c:1371-1445`）
- 父子进程通过 `START_SYNC_*` 消息进行多阶段同步握手
  （`start.c:1364-1547, 2009-2140`）

### 2.4 父子同步时序图

```
       parent(monitor)                    child(container init)
           |                                      |
           +-- clone3() / clone() ---------------->|
           |                                      |
           |<-- START_SYNC_STARTUP ---------------|  (child started)
           |                                      |
           +-- set UID/GID mapping -------------->|
           +-- set cgroup ----------------------->|
           |                                      |
           +-- START_SYNC_POST_CONFIGURE --------->|
           |                                      |  (child sets ns)
           |<-- START_SYNC_CGROUP_LIMITS ---------|
           |                                      |
           +-- set cgroup limits ---------------->|
           +-- START_SYNC_READY_START ------------>|
           |                                      |  (child exec)
           +-- enter lxc_poll() event loop            |
```

### 3.2 配置解析机制

配置解析使用**跳转表**模式（`confile.c:191-260`）：

```c
// 每个 lxc.* 配置键对应三个回调
struct lxc_config_t {
    char *name;
    config_set_cb set;   // 设置值
    config_get_cb get;   // 获取值
    config_clr_cb clr;   // 清除值
};

// 全局跳转表
static struct lxc_config_t config_jump_table[] = {
    { "lxc.rootfs.path",    set_config_rootfs_path,    get_config_rootfs_path,    clr_config_rootfs_path },
    { "lxc.mount.entry",    set_config_mount_entry,    get_config_mount_entry,    clr_config_mount_entry },
    { "lxc.net",            set_config_net,            get_config_net,            clr_config_net },
    // ... 几十个配置项
};
```

### 3.3 配置生命周期

```
lxc_conf_init()       <- alloc and init config obj
    |
    +-- load config file -> config_jump_table[].set() callback
    |
    +-- append_unexp_config_line()  <- save raw text(for clone/save)
    |
    +-- [runtime uses config]
    |
    +-- lxc_conf_free()  <- free all config
```

## 4. 状态管理（state.h/c）

### 4.1 状态枚举

容器可处于以下状态（`state.h:12-22`）：

```
STOPPED → STARTING → RUNNING → STOPPING → STOPPED
                       ↓         ↑
                    FREEZING → FROZEN → THAWED(→RUNNING)
                       ↓
                    ABORTING → STOPPED
```

| 状态 | 说明 |
|------|------|
| `STOPPED` | 已停止 |
| `STARTING` | 正在启动 |
| `RUNNING` | 运行中 |
| `STOPPING` | 正在停止 |
| `ABORTING` | 异常中止 |
| `FREEZING` | 正在冻结 |
| `FROZEN` | 已冻结 |
| `THAWED` | 已解冻（映射到 RUNNING） |

### 4.2 状态通知机制

```
lxc_set_state()                         <- start.c:586-604
  +-- notify via anonymous socketpair
  +-- lxc_serve_state_clients()         <- notify subscribed clients
  |     +-- update handler->state
  |     +-- THAWED special case -> map to RUNNING
  |     +-- send state msg to subscribers
  +-- legacy monitord broadcast
```

### 4.3 辅助函数

- `lxc_state2str()` — 状态枚举转字符串（`state.c:29-42`）
- `lxc_str2state()` — 字符串转状态枚举（`state.c:44-53`）

## 5. 容器 Init 进程（cmd/lxc_init.c, initutils.c）

### 5.1 入口

`lxc_init.c` 做命令行解析后调用 `lxc_container_init()`（`cmd/lxc_init.c:98-168`）。

### 5.2 Init 核心逻辑

真正的 init 逻辑在 `initutils.c:482-710`：

```
lxc_container_init()
  |
  +-- setup signal handling, block signals
  +-- drop self exe ref
  |
  +-- fork() ---------------------> child:exec user command
  |                                  +-- exec(cmd)
  |
  +-- parent: becomes in-container "init" supervisor
        +-- rename proc to "init"
        +-- close inherited FDs(close_range / close_inherited)
        +-- catch SIGTERM/SIGPWR/SIGALRM
        |     +-- SIGTERM -> shutdown / kill child
        |     +-- SIGPWR -> like power off
        +-- wait() loop
              +-- keep PID 1-style "reap orphans"
```

这个设计确保了：
1. 容器内始终有 PID 1 进程负责回收僵尸进程
2. 信号处理语义与真实 init 一致
3. 用户命令在子进程中独立运行
